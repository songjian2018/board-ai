# 场景预设：技术 / 架构选型

## 触发关键词
"用 React 还是 Vue" / "Postgres vs MongoDB" / "微服务还是单体" / "云上还是自建" / "上不上 Kubernetes"

## 推荐角色配置（5 角色 standard mode）
1. 第一性原理派（剥离 hype，看底层约束）
2. 执行派（落地工作量 + 招聘难度 + 维护成本）
3. 数据派（性能/成本/可扩展性的可量化指标）
4. 怀疑派（技术债 + 锁定风险）
5. Steelman 派（构造"选 A"和"选 B"的最强版本）

## 推荐 mode
`standard`（3 轮 + 红队）

## 议题重述格式
`【议题】在 [选项A] / [选项B] / [选项C] 中选择哪个技术方案？`
`【类型】选择型`
`【成本】中-高（技术选型重构成本极高）`

## required_inputs（v0.3.0 强制门禁）

```yaml
required_inputs:
  - 候选技术选项：≥ 2 个（具体到版本/产品名，不接受"前端框架"这种类目级表述）
  - 业务场景规模：当前 QPS / 数据量 / 用户量 + 1-3 年规模预测
  - 团队配置：核心工程师数 + 技术栈熟悉度评估
  - 性能要求：QPS / 延迟 / 数据量 / 可用性 SLA
  - 长期维护成本估计：人力 + 云资源 + 锁定风险（迁移成本）
```

## 输入预处理（兼容旧版口径）
让用户给出：
- 业务场景 + 当前规模 + 1-3 年规模预测
- 团队规模 + 团队当前技术栈熟悉度
- 关键性能要求（QPS / 延迟 / 数据量）
- 预算约束（人力 + 云资源）

## 输出额外要求
- **技术对比矩阵**：性能 / 学习曲线 / 招聘市场 / 生态成熟度 / 锁定风险 / 5 年维护成本
- **决策树**：什么前提下选 A，什么前提下选 B
- **降级路径**：选错了怎么撤回（迁移成本估算）

## 已知陷阱
- **Hype Driven Development**：选最流行的不等于选最合适的
- **简历驱动**：工程师倾向选"对我简历好"的技术
- **过早优化**：选了能撑 10亿 QPS 的方案但当前只有 100 QPS
- **NIH (Not Invented Here)**：自研倾向需要被怀疑派压制
