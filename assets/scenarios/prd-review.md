# 场景预设：PRD 审查

## 触发关键词
"帮我审 PRD" / "这份需求文档有问题吗" / "PRD 找盲点"

## 推荐角色配置（5 角色）
1. 数据派（验证指标定义/目标可量化性）
2. 用户派（采纳门槛/认知成本/AHA 时机）
3. 怀疑派（找隐性假设/失败路径）
4. 执行派（拆分到 sprint 级别/工程债）
5. 未羊视角（思维框架审核 + 跨文档冲突敏感）

## 推荐 mode
`standard`（3 轮 + 红队）

## required_inputs（v0.3.0 强制门禁 — 缺一即拒启动辩论）

```yaml
required_inputs:
  - PRD 全文（文档/链接/粘贴）
  - 目标用户画像（≥1 个 persona，含使用场景描述）
  - 业务指标（北极星 + 辅助指标，必须可量化）
  - 资源约束（团队人月数 / 预算上限 / 时间窗口）
  - 关键依赖（依赖文档清单 / 依赖团队 / 依赖外部 API）
```

## PRD 输入预处理（兼容旧版口径）
1. 让用户上传 PRD 文档或粘贴内容
2. 主持人先做 PRD 结构化提炼：[目标 / 核心指标 / 用户场景 / 功能模块 / 依赖项 / 风险] —— 至少这 6 项
3. 若 PRD 涉及多份依据文档（业务规则手册 / 历史 PRD），先识别冲突候选项

## 议题重述格式
`【议题】PRD <名称> 是否可以进入开发阶段？是否需要重大修改？`
`【类型】审查型`
`【成本】中（已立项，但开发前可调整）`

## 输出额外要求
除标准 verdict 卡外，必须额外输出：
- **冲突候选清单**：PRD 内部前后矛盾、与业务规则手册矛盾的具体条目（before/after 取值建议）
- **指标可观测性评估**：每个核心指标的"数据从哪来 / 多久能看到 / 可信度"
- **采纳风险分级**：用户首次使用流失风险 高/中/低 + 具体路径

## 与其他 skill 联动
- 与 `pm-master` 串联：可先用 pm-master 把 PRD 标准化结构化，再用本 skill 做多视角审查
- 与 `weiyang-perspective` 串联：跨文档冲突审计时，把未羊视角作为 single source of truth 提供方
