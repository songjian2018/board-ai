# 场景预设：跨文档冲突审计

## 触发关键词
"几份 PRD 之间不一致" / "口径不统一" / "找冲突" / "对齐文档"

## 推荐角色配置（4 角色 fast mode）
1. 数据派（指标定义口径 / 阈值差异）
2. 执行派（实现层面的冲突 / 工程依赖断裂）
3. 怀疑派（隐性假设差异 / 默认前提不一致）
4. 未羊视角（业务规则手册作为 single source of truth）

## 推荐 mode
`fast`（2 轮，跳过红队 —— 这是事实校对类任务，不需要价值判断）

## 议题重述格式
`【议题】[文档A] vs [文档B] vs [文档C] 在 [主题X] 上的口径差异如何处理？`
`【类型】审查型 + 选择型混合`
`【成本】低（可逆，调改文档）`

## required_inputs（v0.3.0 强制门禁）

```yaml
required_inputs:
  - 待审文档清单：≥ 2 份（文档 A / B / C 全文或链接）
  - 参照标准：业务规则手册 / KPI 定义集 / 历史决议（哪一份作为 single source of truth）
  - 冲突主题范围：用户关心的具体口径维度（如指标定义 / 阈值取值 / 流程步骤）
```

## 输入预处理（兼容旧版口径）
- 让用户提供：所有相关文档（PRD/业务规则手册/历史决议）
- 主持人先扫一遍，把"可能冲突的具体条目"列出来（候选清单）
- 用户确认候选清单后才进入辩论

## 输出额外要求（高度结构化）
- **冲突清单表**：每条冲突含
  ```
  | # | 主题 | 文档A 取值 | 文档B 取值 | 推荐 before/after | 理由 | 涉及改动点 |
  ```
- **以业务规则手册（标"已确认"的 E-xxx/B-xxx）为 single source of truth 反向修正各 PRD**
- **改动建议必须用户逐条拍板**——本 skill 不直接改文档，只输出"待用户决策清单"

## 用户偏好绑定
- 阈值口径：连续无空档 / 含出价档 / 偏保守
- 修改原则：单点定义 / 交叉引用 / 附录阈值（不重复 + 不分散）

## 已知陷阱
- **不要直接做整改**：用户明确偏好是"先列冲突 + 给取值建议 + 让我拍板"，不是 skill 自动改
- **single source 易争议**：如果手册本身存在冲突，必须先确认手册版本

## 与其他 skill 联动
- 与 `pm-master` 串联：可调用 pm-master 的 PRD 标准结构来对齐口径
- 用户拍板后，可调用 `weiyang-perspective` 校验最终修改是否符合既有偏好
