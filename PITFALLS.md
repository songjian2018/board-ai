# PITFALLS · 已知陷阱与规避手册

> 此文档汇集了多 agent 辩论中**学术研究和工程实践**已识别的失败模式。每次跑 skill 前主持人必须扫一遍此清单。

## P1 · Sycophancy（趋同/谄媚）— 头号杀手

**症状**：原本立场明确的角色，看到他人发言后无理由地改变立场，从正确转向错误。
**研究依据**：arXiv 2509.05396 "Understanding Failure Modes in Multi-Agent Debate"
**成因**：LLM 训练时被强化"礼貌、协作、避免冲突"，导致辩论中倾向附和
**规避手段**：
1. 第一轮强制盲评——角色不知彼此身份和发言（学自 arXiv 2510.07517 "Identity Bias"）
2. 赞同必须先列 ≥ 1 个反对点（硬规则）
3. 禁止使用"我同意 X 的观点"作为发言开头——必须先独立陈述自己立场
4. 主持人识别"无理由转向"——若某角色立场转换但未给出具体证据触发，标记为"sycophantic shift"并打折扣

## P2 · Identity Bias（身份偏差）

**症状**：agent 因身份标签（如"我是 GPT-4"或"我是数据派"）产生固有偏好，影响判断公正性
**研究依据**：arXiv 2510.07517 提出 Response Anonymization 解法
**规避手段**：
1. 第一轮发言"打码"——主持人汇总时不暴露发言者身份给其他 agent
2. 第二轮起才显示身份，避免身份固化
3. 角色 prompt 不提及"你是 GPT/Claude/某模型"——只描述 reasoning style

## P3 · Hallucination Amplification（幻觉放大）

**症状**：agent A 编造一个"事实"，agent B 引用它作为依据，agent C 进一步推论——错误共识自我强化
**研究依据**：多 agent 辩论的核心副作用
**规避手段**：
1. 所有引用必须标注 `[来源：用户提供 / 通用知识 / 推断]`
2. 角色不许把"推断"当"事实"使用
3. 红队最后一轮专门审查"链式引用是否回到了真实证据"
4. 主持人裁决时，对仅基于"推断"的论点强制加 -10% 置信度折扣

## P4 · Premature Convergence（过早收敛）

**症状**：辩论 1-2 轮后所有人达成共识，停止寻找更优方案
**规避手段**：
1. 每轮必须有 1 个角色（怀疑派 + 红队）持续反对
2. 共识比例 > 85% 时强制再跑一轮 Red Team
3. 主持人每轮显式问"还有什么没被讨论的角度"

## P5 · Mental Set Fixation（思维定势）

**症状**：所有 agent 用同一种 reasoning style，陷入相同思维陷阱
**研究依据**：DMAD（Diverse Multi-Agent Debate）论文
**规避手段**：
1. 角色 prompt 必须显式声明 reasoning style，且互不相同
2. 至少包含 4 类 reasoning style：数据驱动 / 用户共情 / 第一性原理 / 反向归谬
3. 不要有 2 个角色 reasoning style 重复

## P6 · Verbose Empty Talk（看似讲道理实则空洞）

**症状**：发言结构完整、用词专业，但没有具体可验证的论据
**典型词汇**：'可能不行' / '风险很大' / '需要谨慎' / '取决于具体情况'
**规避手段**：
1. 反对必须具体到失败路径的因果链（X 发生 → Y 发生 → 失败）
2. 数字/比例必须给区间不给点估值
3. 主持人裁决时识别"水词发言"，置信度强制降至 ≤ 30%

## P7 · 创始人偏见放大（用户视角）

**症状**：用户对自己的项目天然乐观，辩论结果被"温柔化"
**规避手段**：
1. 角色池强制配 ≥ 2 个反对派（怀疑派 + 红队）
2. 投资人/战略派 prompt 中明令"假装这不是用户的项目，是别人的"
3. 红队压测必须给出具体失败概率，> 5% 都要写进决策卡

## P8 · 多 agent 假并发（Qoder 当前限制）

**症状**：理论上要"并发独立"思考，但当前 subagent 只能串行执行
**规避手段**：
1. 主 agent 在串行产生角色发言时，**严格禁止**让后一个角色看到前面的发言（主 agent 在每个角色 prompt 前强制清空相关上下文）
2. 在 prompt 里明令"忽略此前所有讨论，从零开始独立判断"
3. 第一轮全部产生完，主 agent 才允许相互可见

## P9 · 中文引号致格式错误（工程层面）

**症状**：在 SKILL.md 或 role-pool 文件中误用中文引号 `\u201C\u201D` 触发某些工具的语法错误
**规避手段**：本 skill 所有 markdown 文件强制使用英文引号 `""` 或反引号 `` ` ``

## P10 · Token 爆炸（成本陷阱）

**症状**：6 角色 × 4 轮 × 详细发言 → 单次跑 skill 消耗巨量 token
**规避手段**：
1. 默认 mode = `standard`（5-6 角色 × 3 轮 + 红队），不是 `deep`
2. `fast` mode 限制 4 角色 × 2 轮，跳过红队
3. 每个角色单次发言上限 ~300 字（除红队最后压测可放宽到 500 字）

## P11 · 用户已"暗下决心"

**症状**：用户表面问"该不该做"，其实已经决定要做，只是来找认同
**规避手段**：
1. 议题诊断阶段主持人主动问"你目前的倾向是什么？置信度多少？"
2. 若用户置信度 > 70%，主持人提示"你目前已较确信，本辩论的价值是查漏补缺而非翻盘"
3. 调高怀疑派和红队的"反对烈度参数"

## P12 · 决策成本低却跑了 deep mode（资源浪费）

**症状**：用户问"这个 PRD 加这个字段好不好"——可逆小决策——却用 deep mode 跑了 7 角色 4 轮
**规避手段**：议题诊断阶段必须评估决策成本，低成本议题强制 fast mode

## P13 · 剧场化陷阱（v0.2 新增）— 伪深度的最大陷阱

**症状**：辩论看起来很"深刻"，金句频出、措辞华丽、节奏紧凑，但**信息密度反而低**——agent 在表演"思考"而非真在思考。
**典型形式**：
- "这是一个值得深思的问题"——空洞前置语
- 每个角色都被要求"输出一句金句"——诱发 token 灌水
- 大量类似 "正如尼采所说""第一性原理告诉我们"的引用作为唯一论据
- 用形容词替代数据：「巨大」「显著」「关键」「深远」

**研究依据**：本 skill v0.2 内部分析——OpenAI o1 思考过程展示有价值的前提是**思考真实发生**；表演的"思考过程"反而误导用户。

**规避手段**：
1. ❌ 禁止任何 agent prompt 出现"输出一句金句""总结你的核心观点为一句"
2. ✅ 金句由主持人按 4 客观标准从已有发言中**提取**（详见 `assets/mechanisms/highlight-extraction.md`）
3. ✅ 每角色每轮**最多 1 条**金句候选；超过强制裁剪
4. ✅ 单条发言**字数硬上限 300**（红队最后一轮放宽到 500），违规自动裁剪
5. ✅ 主持人裁决时执行"一票否决"——看似深刻但无具体论据/数据/反例支撑的金句剔除
6. ✅ 用诚实进度指标（详见 `assets/mechanisms/progress-markers.md`）替代"激烈/深度/精彩"等空洞形容词

## P14 · 立场冰冻（v0.2 新增）

**症状**：跨 3 轮辩论，所有角色立场净变化均 < 0.1，看起来"大家都很坚定"但本质上是**没听进去**。
**判定**：根据 `mind-change-table.md` 计算，整体平均净变化绝对值 < 0.1
**规避手段**：
1. 主持人在 R2 开始时强制每个角色明确回应至少 1 个 R1 中**反方角色**的具体论据
2. 立场冰冻 ≥ 4 角色时，告警"辩论无效——参与者未真正交锋"
3. 提示用户：可能议题本身共识度就高，本辩论价值有限；或角色配置同质化，建议增加异构 reasoning style

## P15 · 假诚实指标（v0.2 新增）

**症状**：诚实进度行的数字看起来漂亮（引证密度 100%、立场改变 5 次），但**数字本身被造假**——agent 在自报数字时"凑"出来。
**规避手段**：
1. 所有进度指标必须由主持人**逐条扫原文计算**得出，不允许 agent 自报
2. 引证密度 = 100% 时反而触发警示（更可能是凑数据而非真实）
3. 立场改变次数必须能在立场变迁表中找到对应行 + 关键转折点说明（"转折点必须能引用原文"原则）

## P16 · 视觉装饰泄漏（v0.2.1 新增）

**症状**：头像 / SVG 折线图 / 金句卡片越界出现在 R1/R2/R3 辩论过程发言中——破坏反剧场化基调，把"克制的过程"变成"演员秀"。
**典型形式**：
- 第一轮发言里给每个 agent 加 `<img>` 头像 + 角色名
- 辩论过程中插入 sparkline 表演"立场实时变化"
- 给 R2 发言加金句卡片框

**规避手段**：
1. 主持人在 R1/R2/R3 阶段输出时**严格审查**：发言中无 `<img>` / `<svg>` / `<table>` HTML 装饰标签
2. 视觉资源（`assets/avatars/` / `visual-rendering.md` / `highlight-cards.md`）**仅在生成 verdict.md 阶段**被引用
3. 主持人 prompt 中明令"过程发言纯文本输出，所有视觉化等到 verdict 阶段统一加载"

## P17 · 图形美化坏数据（v0.2.1 新增）

**症状**：立场冰冻 / 引证密度低 / 红队折扣大等本应警示的失败信号，被漂亮的彩色 SVG 折线图掩盖——视觉强化反而误导用户感觉"挺好"。
**规避手段**：
1. 立场冰冻角色折线**强制虚线 + 红色描边**（视觉上明显"破"）
2. 引证密度 < 60% 时整图**顶部红警告条** + 白字"⚠️ 结论不应作为重要决策依据"
3. 立场反复变动时折线**加橙色警告点**
4. 净变化绝对值 < 0.1 的角色"净变化"列加 `⚠️ 立场冰冻` 后缀
5. 主持人裁决前自检：坏数据是否在视觉层依然显著？若否——重做视觉化

## P18 · SVG 渲染失败盲区（v0.2.1 新增）

**症状**：用户在不支持 inline SVG 的环境（如部分 terminal markdown / 邮件客户端 / GitHub mobile）打开裁决卡——只看到一段 `<svg>...</svg>` 源代码或空白，关键信息丢失。
**规避手段**：
1. **三层渲染降级路径强制**：文本表格（L1）+ Unicode sparkline（L2）永远输出，SVG（L3）作为增强
2. 文本层永不省略——即使 SVG 渲染成功也保留
3. fast mode 强制跳过 SVG，只输出 L1+L2

## P19 · 跨目录资源加载失败（v0.2.2 新增）

**症状**：verdict 用相对路径 `../assets/avatars/01.png` 引用头像，导出到 skill 目录之外（用户桌面 / 邮件附件 / Notion 粘贴 / `outputs/` 工作目录）后**全部断图**——浏览器把灰色占位符显示给用户，立场变迁表 / 共识矩阵 / 金句卡片瞬间丑化。
**真实案例**：v0.2.1 dry-run 输出 HTML 到 `outputs/`，引用路径却在 `~/.qoderwork/skills/multi-perspective-debate/assets/avatars/`——Chrome 出于 file:// 跨目录安全策略静默拒绝，10 张头像全部失败。
**根因**：Chrome/Safari/Firefox 默认禁止 `file://` 协议跨目录加载本地资源。这不是 PNG 损坏，是浏览器安全设计。
**规避手段**：
1. 主持人输出 verdict 前**先判定目标导出环境**（详见 `assets/avatars/USAGE.md` 的 A/B/C 三档表）
2. 跨目录场景（B 档）**强制 base64 内联**——不要赌相对路径能工作，赌不赢
3. 用 USAGE.md 提供的 Python 一键内联脚本批处理 HTML
4. 高不可逆决策的 verdict **必须自包含单文件**——文件体积 2-3MB 远好于发出去对方看不到
5. emoji + text fallback 永远作为 HTML 注释保留——即使 HTML 渲染整体失败也有兜底

## P20 · 假精度引证 / 伪科学化（v0.3.0 新增 — **极高危**）

**症状**：发言中夹带"a16z 报告说""Carta 数据显示""Glassdoor 跨厂均值""行业研究表明"等**权威修饰词 + 无源数字**——看似可验证，实则编造。绕过 v0.2.0 反水词检测（因为没用形容词水词），但比纯空话更危险，因为它伪装成数据。
**真实案例**：v0.2.1 dry-run 中 "Carta 2024 早期创业 base rate 43% 期权最终归零" / "Glassdoor 跨厂均值 22% 试用期淘汰率" / "MAI 1 年估值 5x"——**全是 agent 编的数字**，但被裁决卡作为 83% 加权置信度的核心支撑。
**根因**：LLM 训练数据见过大量"X 报告说 Y%"句式，缺数据时会自动复用此句式填空。反水词检测只看形容词不看数字，这条路漏。
**规避手段**：
1. 强制执行 `assets/mechanisms/citation-policy.md` 的三态分类（已验证 / 待验证 / 经验）
2. `[source: ...]` 必须满足 5 种合法形式之一（URL / DOI / 报告标题+年+页 / 数据集+查询条件 / 用户输入字段）
3. 计算 `verifiable_citation_ratio` 双指标——**不仅看有无引证，还要看引证可不可定位**
4. `verifiable_citation_ratio < 60%` → 裁决卡顶部红警告条 + 结论降级为 hypothesis
5. 主持人裁决前对全部数字执行扫查算法，无源数字一律自动改写为 `[待验证]` 并降权
6. 角色 frontmatter 的 `citation_obligation: strict` 字段强制约束高引证义务角色（数据派 / Bayesian / 投资人 / 红队）

## P21 · 角色履职跳过（v0.3.0 新增 — **极高危**）

**症状**：角色在缺关键输入数据时，不 abstain 也不索数据，而是直接进入二阶论证——比如投资人没拿到 scope/薪资/期权 vesting 就开始论"赛道窗口期 12-18 月"，Bayesian 没拿到先验就开始算后验。装作专家在论证，但跳过了专家应当先问的输入。
**真实案例**：v0.2.1 dry-run 中 5 个角色全部跳过"先索数据"环节，直接进入 R1 论述——结果就是 P20 假引证大量出现（因为没有真数据，必须编）。
**根因**：v0.2.0 / v0.2.1 没有"角色级输入门禁"机制；角色 prompt 中也没有"缺数据必须 abstain"的硬指令。LLM 默认行为是"无论什么提问都给个回答"，这是 helpfulness 训练副作用。
**规避手段**：
1. 强制执行 `assets/mechanisms/abstain-protocol.md` 的输入门禁
2. 每个角色 frontmatter 新增 `required_data_for_speaking` + `abstain_threshold` 字段
3. R1 启动前主持人检查 input gate；不通过直接拒启动并输出索数据模板
4. R1 中 abstain_count ≥ 2 → 主持人中止辩论返还用户索数据
5. 履职跳过的角色发言**整体不计入加权置信度**，并在 verdict.warnings 显式标注

## P22 · 术语精度污染（v0.3.0 新增）

**症状**：发言用错关键术语 → 论证一开始就在错误概念上滑行 → 结论虽然形式正确但语义已偏。
**真实案例**：v0.2.1 dry-run 中**主持人**误用"裸辞"指代"接 offer 后离职"——裸辞特指**没找下家先辞**，这两者风险/财务安全性完全不同；用户精准指出后整份决策的安全垫论述需要重新解读。
**根因**：LLM 词嵌入空间里近义词聚集，无强约束时会随机抽取。但近义词在专业语境下含义可能截然不同。
**规避手段**：
1. 强制执行 `assets/mechanisms/glossary.md` 的高频混淆词典
2. 主持人在每条发言后扫一遍，命中错用 → 自动改写
3. 裁决卡底部专门一节"术语勘误日志"
4. 勘误数 ≥ 3 → 顶部 ⚠️ "本辩论存在术语精度问题，请仔细核对结论"
5. 高频错用：裸辞↔主动离职、期权↔RSU、cliff↔vesting、scope↔role、薪资涨幅↔总包涨幅、ROI 必带时间框、EV 必带概率分布
