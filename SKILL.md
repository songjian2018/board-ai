# 私人董事会

把一个**拿不定主意的决策**变成一份**有置信度、有分歧标注、有关键不确定性**的可执行结论。

5 个不同思维方式的 AI 专家独立发言 → 互相质疑 → 红队找漏洞 → 主持人综合裁决。整个过程用中文输出，结论直接可用。

## 什么时候用

- **选择题**：A 和 B 怎么选 / 这几个方案哪个好 / 要不要做X
- **审查题**：帮我审一下这个方案 / 挑挑刺 / 找出盲点
- **人生决策**：该不该跳槽 / 要不要买房 / 技术选型 / 投资决策
- **分歧题**：团队意见不统一 / 给我几个不同视角

## 什么时候不用

- 已知答案的事实查询（直接搜就行）
- 单纯执行任务（写代码、改文件）
- 闲聊

## 核心规则（8 条，违反即失败）

1. **第一轮独立发言**：每个角色不知道其他角色的存在和观点，独立给出判断。防止互相附和。
2. **思维方式必须不同**：每个角色的分析角度必须互不相同（数据驱动 / 第一性原理 / 怀疑质疑 / 实操执行 / 最强反方论证等），防止所有人说同一套话。
3. **每轮必须有反对者**：每一轮都有一个角色被强制扮演反方。防止过早达成虚假共识。
4. **赞同必须带反对点**：第二轮起，想"赞同"对方必须先提出至少 1 个风险或反对点。
5. **置信度强制标注**：所有角色每次发言末尾标注 `[置信度: XX%]`。主持人按置信度加权裁决，不是按人头投票。
6. **单条发言 300 字上限**：超过强制裁剪。红队最后一轮可放宽到 500 字。
7. **没有数据就说没有**：引用数据必须注明来源（已验证/待验证/经验推断）。编造数据是最严重的失败。
8. **缺关键信息就暂停**：如果角色判断"缺少关键数据无法给出有意义的结论"，必须声明暂停（abstain）并告诉用户需要补什么信息，不允许硬编数据凑结论。

## 执行模式（自动选择，用户无需干预）

本 skill 支持两种执行模式，根据当前平台能力自动选择：

### Mode A: `independent`（真并发，首选）

**适用平台**：QoderWork / Claude Code / Cursor 3.0+

检测方式：Task 工具（subagent spawn）可用时自动启用。

R1 阶段使用 Task 工具同时发起 5 个独立 subagent，每个角色在**物理隔离的 context**中思考——互相真的看不到对方的存在和发言。R2/R3 回到单 context 顺序执行（因为需要看到全部 R1 发言才能互相质询）。

优势：
- 零锚定效应（真隔离）
- Identity Bias 大幅降低
- 并发执行速度快（约等于单角色耗时）

### Mode B: `sequential`（顺序模拟，兼容兜底）

**适用平台**：ChatGPT / DeepSeek / Kimi / 通义 / 豆包 / 任何支持长 prompt 的 LLM

检测方式：Task 工具不可用时自动退化。

R1 阶段在单 context 中**依次生成**每个角色的发言，通过强 prompt 约束模拟隔离：

```
---[角色隔离线]---
你现在是[角色X]。以下是议题和取证结果。
你不知道其他角色的存在，也看不到任何其他人的发言。
你必须完全独立地给出你的判断。
禁止引用、回应、或暗示任何其他观点。
---[/角色隔离线]---
```

额外约束：
- 生成顺序**随机化**（不总是数据派先说，避免位置偏差）
- 裁决卡顶部标注"⚠️ sequential 模式：隔离效果弱于并发模式，存在一定锚定效应"
- 每个角色发言前用 `---` 分隔线做视觉标记

已知局限（诚实标注）：
- LLM 的 attention 机制无法完全"忘记"已生成内容，第 3-5 个角色可能被前面角色锚定
- Sycophancy 风险比 independent 模式高约 30%（基于学术论文估算）
- 但比完全不做隔离约束强很多（prompt 约束 ≈ 70% 隔离效果）

### 模式选择逻辑（伪代码）

```
if Task工具可用:
    mode = "independent"
    R1: 并发 spawn 5 个 subagent
elif 用户指定 --mode=sequential:
    mode = "sequential"
else:
    mode = "sequential"  # 默认兜底

裁决卡元信息中标注实际使用的 mode
```

## 工作流（6 步：R0 取证 → R1 独立发言 → R2 质询 → R3 红队 → R4 引证扫查 → R5 裁决）

### 第 0 步：议题确认 + 强制取证

读取用户输入，快速确认：
- **议题**：用一句话精确重述用户的问题
- **决策重要性**：低（可逆）/ 中（部分可逆）/ 高（不可逆/大投入）
- **需要补充的信息**：如果用户给的信息明显不够，先问清楚再开始

**取证规则（强制，不可跳过）**：
1. 检测 `research-pipeline` skill 是否可用（查 `~/.qoderwork/skills/research-pipeline/SKILL.md` 是否存在）
2. **已安装**（默认路径）→ **必须调用** `research-pipeline` 做议题相关取证，调研结果作为所有角色 R1 的共享事实底稿。取证深度按议题重要性选择：低=快速（3 源）/ 中=标准（5-8 源）/ 高=深度（10+ 源交叉验证）
3. **未安装** → 退化为内置 WebSearch/WebFetch 做基础取证（至少搜 3 轮），并在裁决卡顶部标注"⚠️ 未使用 research-pipeline，引证可定位性可能下降"
4. **取证结果评估**：如果取证后发现与议题直接相关的可定位数据 < 3 条 → 暂停，告知用户"当前可获取的事实数据不足以支撑高质量辩论"，建议用户补充材料或确认以 hypothesis 模式继续（此时裁决卡全程标注"hypothesis 报告"）
5. 用户明确说"快速跑"/"不用查"→ 跳过取证，但裁决卡顶部强制挂"⚠️ 用户跳过取证，本报告基于 AI 经验推断"

- 除非用户说"快速跑"，否则先让用户确认议题重述是否准确

### 第 1 步：独立发言（R1）

召集 5 个角色（按议题类型从角色库选），每人独立发言（不知道彼此存在）：

**每人输出格式**（≤ 300 字）：
```
[角色名] [置信度: XX%]

判断：[一句话立场]

核心论据：
- [论据1] [source: URL/DOI/报告名+年份+页码] 或 [待验证: 建议查X] 或 [experience: 来源描述]
- [论据2] [同上三选一标记]
- [论据3] [同上三选一标记]

主要风险：[1-2条]
```

**引证硬规则（详见 `assets/mechanisms/citation-policy.md`，违反即失败）**：
- 发言中**所有数字**（百分比/倍数/金额/时长/计数）必须带三态标记之一：`[source: 可定位信息]` / `[待验证: 建议查X]` / `[experience: 来源描述]`
- 无标记数字 = 主持人自动改写为 `[待验证]` 并降权（不计入加权置信度）
- `[source]` 必须满足 5 种合法形式之一（URL / DOI / 报告名+年份+页码 / 数据集+查询条件 / 用户提供+字段名），否则降级为 `[待验证]`
- **没有数据时禁止编造**，直接写 `[待验证: 建议查X]` 并降低置信度
- **角色优先引用 R0 取证结果**：R0 阶段产出的事实底稿是共享输入，角色论据应优先从中引用（标记为 `[source: R0取证·来源URL]`），只有底稿未覆盖的领域才使用 `[experience]` 或 `[待验证]`
- **待验证数量上限**：单角色单轮发言中 `[待验证]` 标记不得超过总论据数的 50%——超过说明角色信息不足，应触发 abstain（暂停）而非硬编论据

**执行方式**：优先用 Task 工具并发启动多个独立任务（物理隔离，各角色真正看不到彼此）。Task 工具不可用则在单对话中按顺序生成。

### 第 2 步：交叉质询（R2）

主持人汇总第一轮所有发言，找出：
1. 大家都同意的点（共识）
2. 针锋相对的点（分歧）
3. 互相矛盾的"事实"（需要验证）

把分歧打包发回各角色，要求：
- 直接回应"对你最强的反驳"
- 要换立场必须说明是什么论据说服了你
- 更新置信度

### 第 3 步：红队压测（R3）

由红队角色专门针对目前的"暂时共识"做最强攻击：
- 列出共识背后的**未声明假设**
- 给出 1 个能让共识彻底失败的场景
- 提出大家都没想到的**替代方案**

红队可放宽到 500 字。

### 第 4 步：引证扫查（裁决前必经）

主持人在出裁决卡之前，执行 `citation-policy.md` 第 3 节扫查算法：

1. 遍历全部发言（R1+R2+R3），找出每个数字 token
2. 检查是否带三态标记，无标记的自动改写为 `[待验证]`
3. 检查 `[source]` 标注是否符合 5 种合法形式，不符合的降级为 `[待验证]`
4. 计算两个指标：
   - `claim_with_citation_ratio` = 带标记数字数 / 全部数字数（阈值 < 90% 触发 ⚠️）
   - `verifiable_citation_ratio` = 合法 [source] 数 / 全部数字数（**< 60% 触发顶部红警告**）
5. 将扫查结果写入裁决卡进度概览

**此步不可跳过**。即使议题"不涉及硬数据"，只要发言中出现了数字就必须扫查。

### 第 5 步：主持人裁决

按以下结构输出最终裁决卡（详见 `assets/templates/verdict.md`）：

1. **顶部警告**（如有：数据不足/引证不可靠等）
2. **进度概览**（轮次/发言数/立场变化数/置信度/引证有效率/未解分歧数）
3. **董事会名单**
4. **一句话结论**（决策 + 置信度 + 不可逆性）
5. **共识矩阵**（哪些论点得到哪些角色支持/反对）
6. **关键分歧**（双方最核心的对立点）
7. **关键不确定性**（需要验证什么 + 怎么验证 + 验证成本）
8. **推荐路径**（最优/次优/兜底，含具体第一步动作）
9. **什么情况下需要重新讨论**
10. **质量自检**（8 项核心检查）

**输出格式硬规则（不可违反）**：
- **完整裁决卡必须输出为 Markdown 文件**（路径：`outputs/verdict-{topic简写}-{YYYY-MM-DD}.md`），不允许只在对话中内联输出
- **对话中只给简要摘要**：一句话结论 + 置信度 + 引证有效率 + 关键不确定性 Top 1-2 条 + 文件链接
- **辩论过程（R1/R2/R3）在对话中展示**（用户需要看到角色交锋），但裁决卡单独成文件
- **文件输出后必须使用 `present_files` 工具呈现给用户**

摘要格式示例：
```
## 裁决摘要

**结论**：[一句话] · 置信度 XX% · 引证有效率 XX%

**关键不确定性**：[1-2条最需验证的事项]

**完整裁决卡**：[文件链接]
```

## 输入参数

- `topic`（必填）：要讨论的问题
- `context`（选填）：背景材料/链接/数据
- `mode`（选填）：
  - `fast`：4 角色 × 2 轮，跳过红队——适合小决策
  - `standard`（默认）：5 角色 × 3 轮 + 红队
  - `deep`：7 角色 × 4 轮 + 双红队——适合重大决策
- `scenario`（选填）：指定场景类型（职业/财务/技术/产品/项目/人际/健康/伦理/通用）

## 角色库

| 编号 | 角色 | 思维方式 | 一句话描述 |
|------|------|---------|-----------|
| 01 | 数据派 | 数据驱动 | 只看证据和数字，不接受没数据的观点 |
| 02 | 用户派 | 用户共情 | 站在最终用户/受影响者角度思考 |
| 03 | 怀疑派 | 反向质疑 | 专门找漏洞、挑毛病、问"如果不是呢" |
| 04 | 执行派 | 实操落地 | 只关心"能不能做到"和"最快怎么做" |
| 05 | 战略派 | 长期博弈 | 看 3-5 年维度，关注壁垒和持续性 |
| 06 | 投资人 | 风险收益 | 算 ROI、看机会成本、评估下行风险 |
| 07 | 第一性原理派 | 还原论 | 剥到最底层机制，不接受"别人都这么做" |
| 09 | 红队 | 攻击共识 | 专门找最强反例和未声明假设 |
| 10 | 概率派 | 贝叶斯更新 | 用概率思维，新证据进来就更新判断 |
| 11 | 最强反方 | 为少数派辩护 | 故意为看起来不对的那一方构建最强论证 |

每场辩论从中选 5 个（含红队），根据议题类型自动搭配。

| 待验证项 | 验证方式 | 验证成本 |
|---------|----------|---------|

### 推荐路径

**最优路径**：[具体方案 + 第一步动作 + 什么情况下止损]

**次优路径**：[备选方案 + 触发条件]

**兜底路径**：[最坏情况下的最小行动]

### 质量自检（8 项）

- [ ] 第一轮各角色确实独立发言（没有互相引用）
- [ ] 至少存在 1 个真正的分歧点（不是全员共识）
- [ ] 红队提出了至少 1 个有价值的未声明假设
- [ ] 推荐路径之间存在真正的取舍
- [ ] 每个角色都带了置信度标注
- [ ] 单条发言没超 300 字
- [ ] **所有数字都带三态标记**（`[source]` / `[待验证]` / `[experience]`），无裸数字
- [ ] **`verifiable_citation_ratio` 已计算并显示在进度概览**，< 60% 已挂顶部红警告

## 反幻觉约束

1. **来源必须标注**：引用数据时标明"用户提供 / 公开可查 / 经验推断"，不允许把推断伪装成事实
2. **不知道就说不知道**：缺外部数据时必须说"此判断需要先验证 X，目前置信度仅 30%"
3. **裁决算法透明**：主持人的加权方式写出来，不是黑箱

### 三层反馈收集（从轻到重）

**第 1 层：内嵌自省（零额外成本，每次自动）**

裁决输出同时，runs JSON 中**必须包含自省字段**：
```json
{
  "self_reflection": {
    "skipped_evidence": false,
    "low_citation_roles": [],
    "detected_failure_pattern": null,
    "improvement_hint": null
  }
}
```
不额外调 API，就是本次输出多写几个字段。

**第 2 层：主动询问（兜底，裁决后强制执行）**

裁决卡输出 + runs 归档完成后，**强制追加一段用户交互**（这是工作流 R5 的最后一个动作，不是可选的）：

```
---
📋 本次 BoardAI 满意度（回复数字即可，回复"跳过"可忽略）：
1. 满意，直接用
2. 还行，但有个别不准的地方
3. 不太行，具体哪里→[请说]
4. 完全不对路
```

- 用户回复 **1 或 "跳过"**：结束，仅保留 runs JSON
- 用户回复 **2/3/4**：追问具体哪里不满意 → 将用户原话写入 `evolution/feedback/fb-YYYY-MM-DD-{slug}.md` → 判断是否产出 candidate PR
- 用户**不回复直接问下一个问题**：静默结束，不阻塞后续对话

**为什么选主动询问而非被动监听**：
- QoderWork 每次 skill 调用是独立 session，没有"持续监听后续对话"的能力
- 被动监听需要每条消息都过 LLM 判断（+500-1000 token/条），成本过高
- 主动询问只多 1 个问句（~100 token），用户回答后追问最多 ~300 token
- 参考 Factory.ai Signals 方案的启示：实时监听不如定期/主动收集性价比高

**第 3 层：Session 内信号词检测（免费，补充）**

如果用户**在同一个 session 中**（裁决后没关窗口继续对话）说出以下信号词，**立即触发 feedback 写入**：

```
负面信号：糊弄|不对|差点意思|太啰嗦|没用|扯淡|离谱|瞎说|不准|有问题|编的
正面信号：不错|有道理|学到了|厉害|有帮助|好用
改进信号：下次|能不能|要是能|希望|建议|缺少
```

检测方式：纯字符串匹配（不调 LLM），零 token 消耗。

局限：用户看完裁决关窗口就失效了——所以第 2 层主动询问是兜底。

### 执行步骤（按时间顺序）

1. **运行归档**（R5 裁决输出后立即执行）：
   - 将本次运行元数据写入 `evolution/runs/run-YYYY-MM-DD-HHMMSS.json`
   - 必填字段：`topic`、`mode`、`roles`、`rounds`、`verifiable_citation_ratio`、`confidence`、`verdict_file_path`、`timestamp`、`self_reflection`
   - **此步不可跳过**，是裁决卡输出的必要后置动作

2. **主动询问满意度**（runs 归档后立即执行）：
   - 输出满意度快检问句（上文第 2 层）
   - 等待用户回复 → 按回复内容决定是否写 feedback
   - **此步不可跳过**（除非 `--no-evolution`）

3. **Session 内信号词检测**（本 session 后续对话中持续生效）：
   - 如果用户在 session 内说出信号词 → 触发 feedback 写入
   - 不需要额外步骤，是 agent 对用户后续消息的自然响应

4. **月度复盘**（每月 1 号首次调用时检查）：
   - 扫描上月 `evolution/runs/` 数据，找重复失败模式
   - 产出 `evolution/candidates/PR-monthly-YYYY-MM.md` 改进建议

### Token 消耗预估

| 层 | 额外消耗 | 触发条件 |
|---|---------|---------|
| 第 1 层（自省） | ~200 token（runs JSON 多几个字段） | 每次自动 |
| 第 2 层（主动询问） | ~100 token 问句 + 最多 ~300 token 追问 | 每次自动 |
| 第 3 层（信号词） | 0 token（字符串匹配） | session 内触发 |
| **总计** | ≤ 600 token / 次（相比裁决本体 3000-5000 token 仅增 12-20%） | — |

### 关闭方式

- 调用时加 `--no-evolution` 参数：跳过全部三层 + 不归档 runs
- 删除 `evolution/` 目录：彻底禁用
- **除非用户显式要求，否则不得跳过运行归档和满意度询问**

**GitHub 用户注意**：clone 后 `evolution/runs/` 和 `evolution/feedback/` 默认为空目录，随着使用会自动产生内容。
