# 私人董事会

把一个**拿不定主意的决策**变成一份**有置信度、有分歧标注、有关键不确定性**的可执行结论。

5 个不同思维方式的 AI 专家独立发言 → 互相质疑 → 红队找漏洞 → 主持人综合裁决。整个过程用中文输出，结论直接可用。

## 什么时候用

- **选择题**：A 和 B 怎么选 / 这几个方案哪个好 / 要不要做X
- **审查题**：帮我审一下这个方案 / 挑挑刺 / 找出盲点
- **人生决策**：该不该跳槽 / 要不要买房 / 技术选型 / 投资决策
- **分歧题**：团队意见不统一 / 给我几个不同视角

## 什么时候不用

- 已知答案的事实查询（直接搜就行）
- 单纯执行任务（写代码、改文件）
- 闲聊

## 核心规则（8 条，违反即失败）

1. **第一轮独立发言**：每个角色不知道其他角色的存在和观点，独立给出判断。防止互相附和。
2. **思维方式必须不同**：每个角色的分析角度必须互不相同（数据驱动 / 第一性原理 / 怀疑质疑 / 实操执行 / 最强反方论证等），防止所有人说同一套话。
3. **每轮必须有反对者**：每一轮都有一个角色被强制扮演反方。防止过早达成虚假共识。
4. **赞同必须带反对点**：第二轮起，想"赞同"对方必须先提出至少 1 个风险或反对点。
5. **置信度强制标注**：所有角色每次发言末尾标注 `[置信度: XX%]`。主持人按置信度加权裁决，不是按人头投票。
6. **单条发言 300 字上限**：超过强制裁剪。红队最后一轮可放宽到 500 字。
7. **没有数据就说没有**：引用数据必须注明来源（已验证/待验证/经验推断）。编造数据是最严重的失败。
8. **缺关键信息就暂停**：如果角色判断"缺少关键数据无法给出有意义的结论"，必须声明暂停（abstain）并告诉用户需要补什么信息，不允许硬编数据凑结论。

## 工作流（4 步 + 裁决）

### 第 0 步：议题确认

读取用户输入，快速确认：
- **议题**：用一句话精确重述用户的问题
- **决策重要性**：低（可逆）/ 中（部分可逆）/ 高（不可逆/大投入）
- **需要补充的信息**：如果用户给的信息明显不够，先问清楚再开始
- **是否需要外部取证**：如果议题涉及具体公司/赛道/技术现状/市场数据，调用 `research-pipeline` 先拉一份实时事实底稿；调研结果作为所有角色 R1 阶段的共享输入
- 除非用户说"快速跑"，否则先让用户确认议题重述是否准确

### 第 1 步：独立发言（R1）

召集 5 个角色（按议题类型从角色库选），每人独立发言（不知道彼此存在）：

**每人输出格式**（≤ 300 字）：
```
[角色名] [置信度: XX%]

判断：[一句话立场]

核心论据：
- [论据1，注明来源]
- [论据2]
- [论据3]

主要风险：[1-2条]
```

**执行方式**：优先用 Task 工具并发启动多个独立任务（物理隔离，各角色真正看不到彼此）。Task 工具不可用则在单对话中按顺序生成。

### 第 2 步：交叉质询（R2）

主持人汇总第一轮所有发言，找出：
1. 大家都同意的点（共识）
2. 针锋相对的点（分歧）
3. 互相矛盾的"事实"（需要验证）

把分歧打包发回各角色，要求：
- 直接回应"对你最强的反驳"
- 要换立场必须说明是什么论据说服了你
- 更新置信度

### 第 3 步：红队压测（R3）

由红队角色专门针对目前的"暂时共识"做最强攻击：
- 列出共识背后的**未声明假设**
- 给出 1 个能让共识彻底失败的场景
- 提出大家都没想到的**替代方案**

红队可放宽到 500 字。

### 第 4 步：主持人裁决

按以下结构输出最终裁决卡（详见 `assets/templates/verdict.md`）：

1. **顶部警告**（如有：数据不足/引证不可靠等）
2. **进度概览**（轮次/发言数/立场变化数/置信度/未解分歧数）
3. **董事会名单**
4. **一句话结论**（决策 + 置信度 + 不可逆性）
5. **共识矩阵**（哪些论点得到哪些角色支持/反对）
6. **关键分歧**（双方最核心的对立点）
7. **关键不确定性**（需要验证什么 + 怎么验证 + 验证成本）
8. **推荐路径**（最优/次优/兜底，含具体第一步动作）
9. **什么情况下需要重新讨论**
10. **质量自检**（8 项核心检查）

## 输入参数

- `topic`（必填）：要讨论的问题
- `context`（选填）：背景材料/链接/数据
- `mode`（选填）：
  - `fast`：4 角色 × 2 轮，跳过红队——适合小决策
  - `standard`（默认）：5 角色 × 3 轮 + 红队
  - `deep`：7 角色 × 4 轮 + 双红队——适合重大决策
- `scenario`（选填）：指定场景类型（职业/财务/技术/产品/项目/人际/健康/伦理/通用）

## 角色库

| 编号 | 角色 | 思维方式 | 一句话描述 |
|------|------|---------|-----------|
| 01 | 数据派 | 数据驱动 | 只看证据和数字，不接受没数据的观点 |
| 02 | 用户派 | 用户共情 | 站在最终用户/受影响者角度思考 |
| 03 | 怀疑派 | 反向质疑 | 专门找漏洞、挑毛病、问"如果不是呢" |
| 04 | 执行派 | 实操落地 | 只关心"能不能做到"和"最快怎么做" |
| 05 | 战略派 | 长期博弈 | 看 3-5 年维度，关注壁垒和持续性 |
| 06 | 投资人 | 风险收益 | 算 ROI、看机会成本、评估下行风险 |
| 07 | 第一性原理派 | 还原论 | 剥到最底层机制，不接受"别人都这么做" |
| 09 | 红队 | 攻击共识 | 专门找最强反例和未声明假设 |
| 10 | 概率派 | 贝叶斯更新 | 用概率思维，新证据进来就更新判断 |
| 11 | 最强反方 | 为少数派辩护 | 故意为看起来不对的那一方构建最强论证 |

每场辩论从中选 5 个（含红队），根据议题类型自动搭配。

## 裁决卡输出规范

### 进度概览（表格）

| 指标 | 值 |
|------|-----|
| 轮次 | 3/3 |
| 总发言数 | 16 |
| 立场变化 | 3 次 |
| 未解分歧 | 2 项 |
| 综合置信度 | 68% |
| 数据可靠度 | 中（部分论据待验证） |

### 共识矩阵

| 论点 | 支持 | 反对 | 加权置信度 |
|------|------|------|-----------|
| [论点1] | 数据派、执行派、概率派 | 怀疑派 | 75% |
| [论点2] | 怀疑派、最强反方 | 数据派、战略派 | 55% |

### 关键不确定性

| 待验证项 | 验证方式 | 验证成本 |
|---------|----------|---------|
| [具体问题] | [具体操作] | [时间/金钱] |

### 推荐路径

**最优路径**：[具体方案 + 第一步动作 + 什么情况下止损]

**次优路径**：[备选方案 + 触发条件]

**兜底路径**：[最坏情况下的最小行动]

### 质量自检（8 项）

- [ ] 第一轮各角色确实独立发言（没有互相引用）
- [ ] 至少存在 1 个真正的分歧点（不是全员共识）
- [ ] 红队提出了至少 1 个有价值的未声明假设
- [ ] 推荐路径之间存在真正的取舍
- [ ] 每个角色都带了置信度标注
- [ ] 单条发言没超 300 字
- [ ] 没有编造数据（所有数字都标了来源）
- [ ] 如有引证不足，已在顶部标注警告

## 反幻觉约束

1. **来源必须标注**：引用数据时标明"用户提供 / 公开可查 / 经验推断"，不允许把推断伪装成事实
2. **不知道就说不知道**：缺外部数据时必须说"此判断需要先验证 X，目前置信度仅 30%"
3. **裁决算法透明**：主持人的加权方式写出来，不是黑箱

## 与其他 skill 的协作（均为可选，不装也能独立运行）

- **取证环节（推荐）**：R0 议题确认时如果议题涉及外部事实（公司/赛道/技术现状），调用 `research-pipeline` 拉取**实时、可定位**的来源数据，避免角色编造。**未安装 research-pipeline 时**：skill 正常运行，退化为使用内置 WebSearch/WebFetch 做基础取证，引证可定位性可能下降。
- PRD 审查 → 可先用 `pm-master` 结构化，再用本 skill 辩论
- 项目立项 → 辩论完可串联 `apex-protocol` 做执行规划
- 数据决策 → 调用 `growth-hunter` 提供数据，本 skill 只做解读+辩论
- 需要新角色 → 用 `nuwa-skill` 蒸馏出新的专家角色卡

> **独立使用说明**：本 skill 零外部依赖即可运行。上述协作 skill 全部为"有则更好、无则正常"的可选增强。

## 自我进化机制（可选启用）

本 skill 内置 evolution 协议（详见 `evolution/EVOLUTION.md`），**默认开启、可随时关闭**。

**它做什么**：
- **每次调用自动归档**到 `evolution/runs/` —— 记录输入/输出/置信度/耗时等元数据，不打扰用户
- **检测到反馈信号词**（"这次不对"/"差点意思"/"希望增加"等）→ 写入 `evolution/feedback/` 并自动产出改进候选（PR 格式）
- **每月初自动月度复盘** → 扫描上月运行数据，找出重复失败模式，产出改进建议
- **改进由用户拍板** —— skill 永远不会自己改自己，所有候选改动需人工审核
- **采纳前必跑回归测试**（`tests/dry-run-cases.md`），测试失败自动回滚

**关闭方式**：调用时加 `--no-evolution` 参数，或删除 `evolution/` 目录。

**GitHub 用户注意**：clone 后 `evolution/runs/` 和 `evolution/feedback/` 默认为空目录，随着使用会自动产生内容。你可以把 `evolution/` 加入 `.gitignore` 仅保留协议文件，或保留全目录追踪自己的使用记录。
